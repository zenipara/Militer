# Gate Pass Validation & Auto-Approval System

## Overview

This document describes the comprehensive validation and automatic approval system for Gate Pass submissions in KARYO OS. The system ensures data integrity while providing intelligent auto-approval for eligible submissions.

## Architecture

### 1. Frontend Validation Layer
**Location:** `src/lib/validation/gatePassValidation.ts`

Provides real-time client-side validation with:
- Field-level validation functions
- Form-level comprehensive validation
- Auto-approval criteria checking
- Status transition validation

### 2. Backend Validation Layer
**Location:** `supabase/migrations/20260421140000_enhance_gatepass_validation_autoapproval.sql`

Server-side RPC functions enforce:
- Business logic validation
- Status transition rules
- Authorization checks
- Auto-approval decision logic

### 3. Frontend Components
**Locations:**
- `src/components/gatepass/GatePassForm.tsx` - Enhanced form with validation feedback
- `src/store/gatePassStore.ts` - Store with approval response handling
- `src/lib/api/gatepass.ts` - API layer with response types

## Validation Rules

### Keperluan (Reason for Leave)
| Rule | Value | Example |
|------|-------|---------|
| Min length | 5 characters | "Menghadiri rapat penting" |
| Max length | 255 characters | - |
| Type | Text | - |
| Required | Yes | - |

### Tujuan (Destination)
| Rule | Value | Example |
|------|-------|---------|
| Min length | 3 characters | "Bandung" |
| Max length | 255 characters | "Kantor Polda Jawa Barat" |
| Type | Text | - |
| Required | Yes | - |

### Waktu Keluar (Departure Time)
| Rule | Value | Explanation |
|------|-------|-------------|
| Not in past | Yes | Must be future datetime |
| Max future | 30 days | Cannot schedule >30 days ahead |
| Format | ISO 8601 | 2026-04-21T10:00:00Z |
| Required | Yes | - |

### Waktu Kembali (Return Time)
| Rule | Value | Explanation |
|------|-------|-------------|
| > Waktu Keluar | Yes | Must be after departure |
| Max duration | 7 days | Cannot exceed 7-day duration |
| Max future | 30 days | Cannot schedule >30 days ahead |
| Format | ISO 8601 | 2026-04-21T17:00:00Z |
| Required | Yes | - |

### QR Token
| Rule | Value | Explanation |
|------|-------|-------------|
| Length | 64 characters | 32 bytes in hex format |
| Format | Hex (0-9, a-f) | Auto-generated by system |
| Unique | Yes | Indexed for fast lookup |
| Required | Yes | - |

## Auto-Approval Criteria

### For Komandan (Commander)
```
✅ ALWAYS Auto-Approved
- Their submissions are inherently trusted
- Treated as pre-approval for their own leaves
```

### For Admin
```
✅ ALWAYS Auto-Approved
- System administrators have full authority
```

### For Prajurit (Soldier)
```
✅ Auto-Approved IF ALL criteria are met:
1. Previous Approvals: ≥ 3
   - Has consistent approval track record
   
2. Destination Type: Repeated/Known
   - Has visited destination before
   - Showing predictable patterns
   
3. Duration: ≤ 1 day (24 hours)
   - Short, manageable leave
   - Lower risk
   
4. Timing: Working Hours
   - Monday-Friday only
   - 7:00 AM - 6:00 PM
   - Excludes weekends and after-hours

❌ Otherwise: Pending Komandan Approval
- Requires manual review and approval
```

### Example Scenarios

#### Scenario 1: Prajurit Auto-Approved
```
Soldier: Rifka (previous 5 approvals)
Destination: Bandung (visited 3 times before)
Duration: 9:00-17:00 (8 hours)
Day: Wednesday
Status: ✅ AUTO-APPROVED
Reason: Good history + known destination + short duration + working hours
```

#### Scenario 2: Prajurit Needs Approval
```
Soldier: Budi (previous 1 approval)
Destination: Jakarta (first time)
Duration: 2 days
Status: ⏳ PENDING
Reason: New destination + long duration + insufficient history
```

#### Scenario 3: Prajurit Auto-Rejected (Missing Criteria)
```
Soldier: Andi (previous 4 approvals)
Destination: Surabaya (visited 2x before)
Duration: 3 days ❌ (exceeds 24h limit)
Status: ⏳ PENDING
Reason: Duration > 1 day disqualifies auto-approval
```

## Status Transition Flow

```
┌─────────────────────────────────────────────────────────────┐
│                    GATE PASS LIFECYCLE                      │
└─────────────────────────────────────────────────────────────┘

                     ┌─────────────┐
                     │   CREATE    │
                     │    FORM     │
                     └──────┬──────┘
                            │
                    ┌───────▼────────┐
                    │  VALIDATION    │
                    │   & APPROVAL   │
                    └───────┬────────┘
                            │
                ┌───────────┼───────────┐
                │           │           │
         [PENDING]   [APPROVED]*   [REJECTED]
         (Need OK)   (Auto/Manual)  (Denied)
                │           │
                │    ┌──────▼──────┐
                │    │ SCAN KELUAR  │
                │    │   AT POS     │
                │    └──────┬───────┘
                │           │
                │    [CHECKED_IN]
                │      (OUT)
                │           │
                │    ┌──────▼──────┐
                │    │ SCAN KEMBALI │
                │    │   AT POS     │
                │    └──────┬───────┘
                │           │
                │    [COMPLETED]
                │    (RETURNED)
                │
            ┌───────────────┐
            │ OVERDUE CHECK │
            │ (waktu kembali│
            │   < now)      │
            └───────┬───────┘
                    │
                    ▼
              [OVERDUE]
              (ALARM!)
```

### Valid State Transitions

| From | To | By | Notes |
|------|----|----|-------|
| pending | approved | Admin/Komandan | Manual review |
| pending | rejected | Admin/Komandan | Approval denied |
| approved | checked_in | Guard | Scan exit at post |
| checked_in | completed | Guard | Scan return at post |
| checked_in | overdue | System | Auto-detect (waktu_kembali < now) |
| overdue | completed | Guard | Late scan return |

### Invalid Transitions (Rejected)
```
pending → completed        ❌ Must go through approved first
approved → rejected        ❌ Already approved
checked_in → pending       ❌ Cannot revert status
completed → any            ❌ Final state, immutable
```

## Backend Decision Logic

### Phase 1: Data Validation
```sql
-- Validate and reject obvious bad data early
CHECK (waktu_kembali > waktu_keluar)
CHECK (LENGTH(tujuan) BETWEEN 3 AND 255)
CHECK (LENGTH(keperluan) BETWEEN 5 AND 255)
CHECK (EXTRACT(EPOCH FROM (waktu_kembali - waktu_keluar)) <= 7 DAYS)
```

### Phase 2: Role-Based Logic
```sql
IF role = 'komandan' THEN
  status = 'approved'
  auto_approved = TRUE
ELSIF role = 'admin' THEN
  status = 'approved'
  auto_approved = TRUE
ELSIF role = 'prajurit' THEN
  -- Check criteria
  IF has_3_previous_approvals
     AND is_repeated_destination
     AND duration <= 24_hours
     AND is_working_hours THEN
    status = 'approved'
    auto_approved = TRUE
  ELSE
    status = 'pending'
    auto_approved = FALSE
  END IF
END IF
```

### Phase 3: Approval Logging
```sql
INSERT INTO gate_pass_approval_log (
  gate_pass_id,
  approver_id,
  approval_status,
  is_auto,
  criteria_met
) VALUES (...)
```

## API Endpoints

### Insert Gate Pass (with Auto-Approval)
```typescript
POST /rpc/api_insert_gate_pass

Input:
{
  p_user_id: UUID
  p_caller_role: 'prajurit'|'komandan'|'admin'
  p_keperluan: string         // 5-255 chars
  p_tujuan: string            // 3-255 chars
  p_waktu_keluar: timestamp   // future
  p_waktu_kembali: timestamp  // > waktu_keluar, <= 7 days
  p_qr_token: string          // 64 hex chars
}

Output:
{
  gate_pass_id: UUID
  auto_approved: boolean
  status: 'approved' | 'pending'
  approval_reason: string
}

Errors:
- 'gate_pass feature is disabled'
- 'Unauthenticated'
- 'Unauthorized: tidak dapat membuat gate pass untuk orang lain'
- 'Durasi izin maksimal 7 hari'
- 'Tujuan harus 3-255 karakter'
- 'Keperluan harus 5-255 karakter'
```

### Update Gate Pass Status (with Validation)
```typescript
POST /rpc/api_update_gate_pass_status

Input:
{
  p_caller_id: UUID
  p_caller_role: 'admin'|'komandan'|'guard'
  p_id: UUID                  // gate pass ID
  p_status: status            // new status
  p_approved_by: UUID         // optional
  p_approval_reason: string   // optional
}

Output:
{
  gate_pass_id: UUID
  status: string
  message: string
}

Validation Errors:
- Status transitions checked strictly by current status
- Role-based access control enforced
- Komandan can only approve within satuan
```

## Authorization & Security

### Role-Based Access Control

| Role | Can Create | Can Approve | Can Scan | Auto-Approve |
|------|-----------|-------------|----------|--------------|
| Admin | ✅ Any | ✅ Any | ✅ Any | ✅ Always |
| Komandan | ✅ Self | ✅ Satuan | ✅ HQ | ✅ Always |
| Guard | ❌ No | ❌ No | ✅ Post | ❌ Never |
| Prajurit | ✅ Self | ❌ No | ❌ No | ✅ Conditional |

### Satuan-Based Authorization
- Komandan can only approve gate passes within their satuan
- Guards can scan at any post
- Cross-satuan approvals rejected

## Troubleshooting

### Issue: "Durasi izin maksimal 7 hari"
**Solution:** Choose return time within 7 days of departure

### Issue: "Status transition tidak valid"
**Solution:** Check current status, ensure valid next status

### Issue: Gate pass not auto-approved despite good history
**Solution:** Check all 4 criteria:
1. ≥ 3 previous approvals
2. Repeated destination
3. ≤ 24 hour duration
4. Working hours (Mon-Fri, 7AM-6PM)

### Issue: Cannot scan because "Status adalah pending"
**Solution:** Wait for komandan approval or check auto-approval criteria
